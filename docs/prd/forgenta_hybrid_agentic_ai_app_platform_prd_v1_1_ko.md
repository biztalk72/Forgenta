# Forgenta: Hybrid Agentic AI - App Platform PRD v1.1

## 문서 개요

본 문서는 기존 AIU 스타일 사내 생성형 AI 플랫폼 PRD를 **Forgenta: Hybrid Agentic AI - App Platform**이라는 제품명으로 재정의하고, MVP 개발이 가능한 수준까지 기능 범위와 시스템 요구사항을 구체화한 통합 PRD다.[page:1][web:23] Forgenta는 내부 데이터 기반 생성형 AI, 프롬프트 정제, 에이전트 및 앱 재사용, 멀티모달 응답, 거버넌스 중심 운영을 하나의 플랫폼으로 제공하는 것을 목표로 한다.[page:1][page:3][web:23]

버전 1.1의 핵심 방향은 세 가지다. 첫째, 사용자가 더 좋은 입력으로 시작하도록 프롬프트를 정제하고 유사한 기존 구현 사례를 즉시 탐색하게 한다.[page:2] 둘째, 이미 개발된 앱과 에이전트를 카탈로그 형태로 탐색하고 선택 후 사용, 편집, 삭제, 이동할 수 있게 한다.[web:23][web:25] 셋째, 챗/프롬프트 응답이 텍스트에 머물지 않고 표, CSV, 이미지, SVG, 2D/3D 차트로 확장되는 멀티모달 출력 플랫폼이 된다.[page:4][page:5]

## Step 1: 간단한 설명

Forgenta를 쉽게 비유하면, “회사 안의 AI 운영체제 + 사내 App Store + 데이터 시각화 워크벤치”다.[web:23] 사용자는 질문을 던지는 데서 끝나지 않고, 이미 만들어진 앱이나 에이전트를 찾아 실행하고, 복제해 수정하고, 자신만의 워크플로우로 발전시킬 수 있어야 한다.[web:23][web:25]

또한 답변도 단순 문장만 나오면 안 된다. 기업 업무에서는 숫자는 표로, 시각 정보는 이미지로, 구조도는 SVG로, 추세는 2D/3D 차트로 받아야 실제 업무에 바로 쓸 수 있기 때문이다.[page:4][page:5]

## Step 2: 혼란 지점 확인

이 제품에서 흔히 헷갈리는 지점은 세 가지다.

- “챗봇”과 “하이브리드 앱 플랫폼”은 다르다. Forgenta는 질문-응답 인터페이스를 제공하지만, 본질은 에이전트·앱·데이터 자산을 검색하고 재사용하는 운영 플랫폼이다.[page:1][web:23]
- “멀티모달 입력”과 “멀티모달 응답”은 다르다. MVP 우선순위상 입력은 텍스트 중심으로 시작할 수 있지만, 응답은 반드시 텍스트 외에 표, CSV, 차트, 이미지, SVG 같은 복수 포맷을 지원해야 한다.[page:4][page:5]
- “SVG 생성”은 단순 텍스트 출력이 아니라 보안과 렌더링 정책이 필요한 기능이다. SVG는 XML 기반 벡터 형식이므로 웹 표준과 확장성에는 유리하지만, 스크립트 삽입 등 보안 위험이 있어 sanitization이 필요하다.[page:5]

## Step 3: 정제 사이클

### 제품 비전

Forgenta의 목표는 직원이 AI를 단순 질의응답 도구로 쓰는 수준을 넘어, 내부 문서와 데이터에 근거한 반복 가능한 업무 보조자, 에이전트, 미니 애플리케이션을 직접 구성하고 재사용할 수 있게 하는 것이다.[page:1][web:23] 따라서 제품은 단일 챗봇이 아니라 데이터 레이어, AI 실행 레이어, 프롬프트 자산 레이어, 카탈로그 레이어, 멀티모달 렌더링 레이어, 거버넌스 레이어를 함께 갖춘 엔터프라이즈 AI 운영체계로 설계되어야 한다.[page:2][page:3][web:23]

비유하면 이 제품은 “회사 전용 지식을 담은 안전한 공작소”이면서 동시에 “검증된 앱과 에이전트를 진열한 사내 마켓”이다.[page:1][web:23] 레이크하우스는 재료 창고, ETL과 벡터화 파이프라인은 전처리 라인, 프롬프트 정제 엔진은 작업 지시문 편집기, 카탈로그는 재사용 가능한 완제품 진열장, 멀티모달 응답 엔진은 결과물을 사람 눈에 맞게 번역하는 프레젠테이션 계층이다.[page:2][page:3][page:4]

### 제품 목표

#### 비즈니스 목표

- 내부 데이터 기반 생성형 AI 활용 표준 환경 구축.[page:1]
- 현업 사용자가 템플릿 기반 에이전트와 앱을 직접 실행·수정·공유할 수 있는 셀프서비스 체계 구축.[page:1][web:23]
- 제조, HR, finance 도메인에서 공통적으로 재사용 가능한 데이터 파이프라인과 AI 워크플로우 확보.[page:3]
- 보안, 감사, 승인, 권한 통제를 전제로 한 엔터프라이즈 AI 운영 기반 마련.[web:23]
- 프롬프트와 에이전트 및 앱을 운영 자산으로 표준화해 중복 개발을 줄이고 재사용률을 높임.[page:2][web:23]
- 텍스트 중심 AI를 멀티모달 응답 플랫폼으로 확장해 실제 업무 활용도를 높임.[page:4][page:5]

#### 사용자 목표

- 사용자는 회사 문서와 데이터에 근거한 답변을 받을 수 있어야 한다.[page:1][page:3]
- 사용자는 자유 입력 후 정제된 프롬프트와 유사 사례를 추천받아 더 나은 시작점을 얻어야 한다.[page:2]
- 사용자는 대시보드에서 앱/에이전트 카탈로그를 탐색하고, 선택 후 사용·편집·삭제·이동을 빠르게 수행할 수 있어야 한다.[web:23][web:25]
- 사용자는 필요에 따라 CSV, 표, 이미지, SVG, 2D/3D 차트 형태의 결과를 받아야 한다.[page:4][page:5]
- 관리자는 누가 어떤 데이터로 어떤 프롬프트와 앱/에이전트를 실행했는지 감사 가능해야 한다.[web:23]

## MVP 범위 정의

### MVP에 포함되는 핵심 범위

| 영역 | MVP 포함 내용 |
|---|---|
| 대화/질의 | 챗 워크스페이스, 프롬프트 정제, 유사 사례 추천 |
| 데이터 | 오픈·공유 제조업 데이터, synthetic HR, synthetic finance 데이터를 레이크하우스에 저장 |
| 파이프라인 | ETL/ELT, 문서 파싱, 청킹, 임베딩 생성, 벡터 인덱싱 |
| 검색 | 문서 검색, 유사 프롬프트/에이전트/앱 검색, 하이브리드 RAG |
| 카탈로그 | App Catalog + Agent Catalog, 상세 보기, 사용, 편집, 삭제, 이동 |
| UX 패턴 | 대시보드 내 팝업 또는 사이드 롤업/다운 패널 기반 조작 UI |
| 멀티모달 응답 | 표, CSV 다운로드, 이미지 뷰잉/다운로드, SVG 생성/보기, 2D/3D 차트 |
| 거버넌스 | SSO 전제, RBAC, 실행 로그, 프롬프트 로그, 승인 상태, 소유자 관리 |

### MVP 제외 범위

- 실시간 스트리밍 기반 초저지연 이벤트 처리.
- 완전 자율형 고위험 액션 에이전트.
- 프로덕션 ERP/MES 쓰기 작업 자동 실행.
- 음성 인터페이스와 영상 기반 입력.
- 고급 멀티테넌시 및 외부 고객용 SaaS 기능.

## 대상 사용자

### 1. 일반 현업 사용자

문서 요약, 보고서 초안, 규정 검색, 데이터 질의, 차트 생성 같은 일상 업무를 빠르게 처리하려는 사용자다.[page:1][page:4] 이 사용자는 챗 UI와 검증된 카탈로그 항목을 주로 사용하며, 복잡한 빌더 대신 쉬운 입력 폼과 결과 검토 경험을 원한다.[page:1][web:23]

### 2. 시민개발자 / 파워 유저

자신의 팀 업무에 맞춰 프롬프트, 데이터 소스, 출력 구조를 조합해 앱 또는 에이전트를 만들고 재사용하려는 사용자다.[page:1][web:23] 카탈로그 기반 복제 방식과 노코드·로우코드 편집기가 이 사용자층의 생산성을 끌어올리는 핵심이다.[web:23][web:25]

### 3. 플랫폼 운영자 / 관리자

데이터 적재, 권한 관리, 모델 정책, 멀티모달 렌더링 정책, 프롬프트/응답 감사, 품질 평가를 담당한다.[web:23][page:5] SVG와 파일 출력은 보안 검토와 샌드박싱 정책이 필요하므로 운영자 관점의 제어 기능이 중요하다.[page:5]

## 핵심 사용자 시나리오

### 시나리오 A: 제조 문서 검색과 보고서 초안 작성

사용자는 설비 점검 기준, 작업 표준서, 과거 리포트를 검색해 현재 작업에 필요한 체크리스트와 요약 보고서를 생성한다. 시스템은 관련 문서를 권한 범위 내에서 검색하고, 근거 문서와 함께 초안을 생성해야 한다.[page:1][page:3]

### 시나리오 B: HR/finance synthetic 데이터 기반 표·차트 응답

사용자는 synthetic HR 또는 finance 데이터에 대해 자연어로 질문하고, 시스템은 정형 데이터 질의 결과를 표와 CSV로 제공하며, 필요 시 2D 또는 3D 차트로 함께 렌더링해야 한다.[page:4]

### 시나리오 C: 유사 사례 탐색 후 복제 실행

사용자는 자유 입력 프롬프트를 입력하고, 시스템은 유사한 기존 프롬프트와 앱/에이전트 예제를 추천한다.[page:2] 사용자는 대시보드 패널에서 해당 예제를 선택해 바로 실행하거나, 복제 후 일부를 수정해 자신의 버전으로 저장해야 한다.[web:23][web:25]

### 시나리오 D: 카탈로그에서 앱/에이전트 관리

사용자는 대시보드에서 카탈로그를 열고 앱 또는 에이전트 카드를 탐색한다. 특정 항목을 클릭하면 팝업 또는 사이드 롤업/다운 패널에서 상세 정보와 함께 사용, 편집, 삭제, 이동 액션을 수행해야 한다.[web:23][web:25]

### 시나리오 E: 이미지·SVG 중심 응답 생성

사용자는 공정 흐름, 조직도, 시스템 구성도, 기술 도식화를 요청하고 시스템은 SVG 기반 결과를 생성해 미리보기와 다운로드를 제공해야 한다.[page:5] 또한 문서 기반 응답에서 관련 이미지를 함께 표시하고 다운로드할 수 있어야 한다.[page:4]

## 기능 요구사항

### 1. 챗 워크스페이스 및 프롬프트 정제

- 자연어 질의 입력.
- 근거 문서 및 인용 스니펫 표시.[page:3]
- 결과 복사, 다운로드, 피드백 기록.
- 세션 단위 대화 기록.
- 사용자 권한 범위 내 데이터만 검색.
- 실행 전 정제 프롬프트 및 추천 예제 패널 노출.[page:2]
- 원문 프롬프트와 정제 프롬프트 비교 UI 제공.[page:2]

### 2. 데이터 레이어

Forgenta의 데이터 레이어는 오픈·공유된 제조업 관련 데이터와 synthetic HR, synthetic finance 데이터를 저장하는 데이터 레이크하우스를 포함해야 한다. 레이크하우스는 구조화, 반정형, 비정형 데이터를 한 환경에서 저장·관리하고 거버넌스를 제공하는 기반으로 적합하다.[page:3]

수집 대상은 다음과 같다.

- 제조업 공개 데이터: 설비, 품질, 안전, 공급망, 표준 문서, 산업 리포트 등 공개 가능 데이터.
- Synthetic HR 데이터: 직원 마스터, 부서, 평가, 채용, 교육, 이직, 근태 등 개인정보 비식별 대체 데이터.
- Synthetic finance 데이터: 전표, 비용센터, 예산, 실적, 구매, 지급, 프로젝트 재무 데이터의 합성본.
- 내부 테스트 문서: 정책 문서, 표준 작업서, FAQ, 보고서 샘플.

### 3. ETL/ELT 및 벡터화 파이프라인

- 소스 등록: 파일 업로드, 오브젝트 스토리지, DB 커넥터, API.
- 스키마 프로파일링: 컬럼 타입, 결측치, 유효성 검사.
- 데이터 정제: 포맷 정규화, 중복 제거, 마스킹, 언어/문서 타입 감지.
- 문서 파싱: PDF, DOCX, HTML, CSV, XLSX 텍스트 추출.
- 청킹: 문단/섹션/표 기반 chunk 분리.
- 임베딩 생성: 청크 단위 벡터 생성 및 메타데이터 부착.
- 인덱싱: 벡터 DB 및 키워드 검색 인덱스 동시 적재.
- 재색인: 문서 변경 시 재임베딩 및 버전 관리.

벡터 데이터베이스는 임베딩 저장과 유사도 검색에 적합하고, 레이크하우스는 분석과 데이터 준비의 기반이 되므로 두 레이어를 함께 써야 한다.[page:3]

### 4. 검색 및 추천

- 키워드 검색 + 벡터 검색 + 메타데이터 필터 기반 하이브리드 검색.
- 문서 권한 필터 적용.
- 답변 시 출처 문서, 제목, 스니펫, 버전, 데이터셋 표시.
- 프롬프트 본문, 설명, 실행 이력 요약을 대상으로 한 유사 프롬프트 검색.[page:2]
- 에이전트/앱 메타데이터를 포함한 유사 사례 검색.[web:23]
- 실행 전 유사 예제 3~5개 추천.[page:2]

### 5. 카탈로그 대시보드 UI

중앙화된 에이전트 스토어는 검색, 표준화, 버전 관리, 수명주기 관리, 재사용을 가능하게 하는 핵심 구성요소다.[web:23] Forgenta MVP는 이를 대시보드 핵심 요소로 제공해야 하며, 사용자는 개발된 앱 또는 에이전트를 카탈로그 형태로 보고 바로 조작할 수 있어야 한다.[web:23][web:25]

필수 UI 기능은 다음과 같다.

- App Catalog와 Agent Catalog 두 가지 뷰 제공.[web:23]
- 카드형 리스트 + 필터/검색 + 정렬(추천순, 최신순, 사용량순).
- 카드 선택 시 상세 패널을 **팝업 모달** 또는 **사이드 롤업/다운 패널**로 표시.
- 상세 패널 내 액션: Use, Edit, Delete, Move.
- Move는 카테고리 변경, 소유 팀 변경, 상태 이동(Draft → Review → Active)을 포함.
- Delete는 soft delete 기본, 관리자만 hard delete 가능.
- 상세 패널에서 샘플 입력/출력, 사용 데이터셋, 보안 범위, 소유자, 버전 표시.[web:23]

### 6. 에이전트/앱 수정·복제·실행

ServiceNow는 기존 에이전트를 복제한 뒤 이름과 세부 구성을 수정하는 흐름을 제공하고, 복제본 이름에 기본 “(Copy)” 접미사를 붙인다.[web:25] Forgenta도 같은 수준의 복제 경험을 제공해야 하며, 원본 보존과 파생 계보 추적이 중요하다.[web:25]

필수 동작은 다음과 같다.

- 상세 보기: 목적, 입력, 출력, 데이터 소스, 도구, 권한, 실행 로그 요약 표시.[web:23]
- Use: 해당 앱 또는 에이전트를 즉시 실행하는 진입점 제공.
- Edit: 프롬프트, 변수, 데이터 연결, 출력 형식, 가드레일 편집.
- Clone: 원본 설정을 복사해 새 이름으로 저장하고 필요한 부분만 변경.[web:25]
- Delete: 소유 권한과 승인 정책에 따라 삭제.
- Move: 폴더/카테고리/소유 팀/상태를 변경.
- Compare: 원본과 수정본의 프롬프트, 도구, 데이터 소스, 성능 차이 비교.

### 7. 멀티모달 응답 엔진

Forgenta MVP는 텍스트 중심 답변을 넘어서 복수 포맷 응답을 지원해야 한다. OpenAI의 코드 생성 가이드는 GPT-5.4와 코딩 에이전트 계열 모델이 프런트엔드와 코드 생성에 강하며, 코드·구조화 출력·도구 결합 작업에 적합하다고 설명한다.[page:4] 또한 SVG는 XML 기반 벡터 형식으로서 웹 표준 적합성과 무손실 확장성 측면에서 유리하지만, 생성·렌더링 과정에서 보안 검증이 필요하다.[page:5]

MVP 필수 응답 타입은 다음과 같다.

- Rich text: 서술형 응답, 목록, 근거 문서 링크.
- Table: 정형 응답을 렌더링 가능한 표로 표시.
- CSV export: 표 데이터를 CSV 파일로 다운로드 가능하게 제공.[page:4]
- Image viewing: 응답 내 이미지 썸네일/확대 보기 제공.
- Image download: PNG/JPG/WebP 다운로드 지원.
- SVG generation: 다이어그램, 아이콘, 간단한 구조도, 기술 일러스트 생성 및 다운로드.[page:5]
- 2D charts: line, bar, area, pie, scatter 등.
- 3D charts: 3D scatter, 3D surface, 3D bar 계열의 최소 지원 세트.

### 8. SVG 생성 및 외부 서비스 연동 정책

SVG는 내부적으로 직접 생성할 수도 있고, 외부 서비스와 연동할 수도 있다. 웹 표준 관점에서 SVG는 반응형 스케일링과 정밀한 렌더링에 강점이 있고, 서버사이드 또는 프로그램 방식 생성에도 적합하다.[page:5] 그러나 SVG는 스크립트 삽입, 외부 참조, XSS 같은 보안 위험이 있어 표시 전 sanitization과 허용 태그 화이트리스트가 필요하다.[page:5]

MVP 정책은 다음과 같이 정의한다.

- 기본 원칙: **내부 생성 우선, 외부 서비스는 선택적 확장**.
- 1차 방식: 텍스트 기반 다이어그램과 차트는 내부 코드 생성 또는 서버사이드 SVG 렌더링으로 처리.[page:4][page:5]
- 2차 방식: 고급 일러스트형 SVG가 필요할 경우 외부 서비스 연동 후보를 검토하되, 보안·저작권·비용·감사 로그 확인 후 허용.
- 렌더링 전 sanitization 필수.
- 다운로드용 원본 SVG와 화면용 sanitized SVG를 분리 저장.

### 9. 차트 및 시각화

- 2D 차트는 MVP 필수 기능이다.
- 3D 차트는 분석형 질문에서 선택적으로 활성화하되 최소 3종 제공.
- 차트는 데이터셋 메타데이터와 질의 결과를 연결해 자동 추천 가능해야 한다.
- 사용자는 차트를 이미지 또는 SVG로 저장할 수 있어야 한다.[page:5]
- 차트는 대시보드 카드, 상세 패널, 챗 응답 내 인라인 표시를 모두 지원해야 한다.

### 10. 이미지 뷰잉 및 다운로드

- 응답 이미지 썸네일, 확대 뷰어, 원본 다운로드 제공.
- 이미지 메타데이터: 생성 시각, 생성 프롬프트, 연관 데이터셋, 생성자 저장.
- 카탈로그 앱/에이전트가 생성한 대표 이미지 미리보기 지원.

### 11. 거버넌스 및 운영

- SSO/사내 계정 연동.
- RBAC 및 데이터셋 단위 접근통제.[web:23]
- 프롬프트, 응답, 검색 문서, 모델 사용 로그 저장.[web:23]
- 앱/에이전트별 오너 지정.[web:23]
- 위험 등급별 승인 정책.
- 관리자용 데이터셋/앱/에이전트/사용량 콘솔.
- Clone lineage 및 배포 이력 추적.[web:25]
- SVG 및 파일 다운로드에 대한 감사 로그 저장.[page:5]

## Step 4: 이해도 도전

이 PRD의 핵심을 이해했는지 확인하려면 이 질문에 답해보면 된다. “왜 Forgenta는 챗 UI만으로 충분하지 않은가?” 좋은 답은, 기업에서는 질문 자체보다도 검증된 앱/에이전트의 재사용, 멀티모달 결과물, 보안 통제, 카탈로그 기반 운영이 실제 가치 창출의 핵심이기 때문이다.[web:23][web:25]

또 다른 질문은 이것이다. “왜 SVG 생성은 기능 추가만이 아니라 정책 문제인가?” 답은, SVG가 강력한 벡터 포맷이지만 XML 기반이어서 렌더링 전 sanitization과 보안 통제가 반드시 필요하기 때문이다.[page:5]

## Step 5: 가르침 스냅샷

Forgenta는 단순한 사내 챗봇이 아니다. **더 좋은 프롬프트로 시작하고, 이미 만들어진 앱과 에이전트를 카탈로그에서 재사용하며, 결과를 표·CSV·이미지·SVG·차트로 받아 실제 업무에 바로 쓰게 만드는 하이브리드 Agentic AI-App 플랫폼**이다.[page:2][web:23][web:25][page:4][page:5]
