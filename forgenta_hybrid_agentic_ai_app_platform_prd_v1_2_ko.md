# Forgenta: Hybrid Agentic AI - App Platform PRD v1.2

## 문서 개요

본 문서는 기존 AIU 스타일 사내 생성형 AI 플랫폼 구상을 **Forgenta: Hybrid Agentic AI - App Platform**이라는 신규 제품명으로 재정의하고, 실제 MVP 개발이 가능한 수준까지 요구사항을 구체화한 제품요구사항(Product Requirements Document)이다.[page:1][web:23] 제품의 핵심 방향은 내부 데이터 기반 생성형 AI, 노코드·로우코드형 에이전트/앱 제작, 중앙 카탈로그 기반 재사용, 그리고 운영 가능한 거버넌스를 하나의 대시보드형 플랫폼에 통합하는 것이다.[page:1][web:23][web:35]

v1.2에서 Forgenta는 단순한 사내 챗봇이 아니라, 프롬프트 정제, 유사 사례 검색, 에이전트/앱 카탈로그, 멀티모달 응답, 그리고 대시보드 기반 운영 UI를 갖춘 **하이브리드 에이전틱 AI 앱 플랫폼**으로 정의된다.[page:2][web:23] “Hybrid”라는 표현은 정형 데이터 질의, 문서 기반 RAG, 앱형 워크플로우, 멀티모달 출력, 사람 승인 기반 운영이 한 제품 안에서 함께 동작한다는 뜻으로 사용한다.[page:3][web:35]

## 제품 비전

Forgenta의 목표는 직원이 AI를 단순 대화 도구로 쓰는 수준을 넘어, 내부 문서와 구조화 데이터에 근거해 반복 가능한 업무용 앱과 에이전트를 검색·선택·복제·수정·실행할 수 있게 하는 것이다.[page:1][page:2][web:23] 즉, 제품은 “회사 전용 지식 공작소”이면서 동시에 “검증된 업무용 AI 자산을 배포하는 운영 플랫폼”이어야 한다.[page:1][web:23]

비유하면 Forgenta는 세 가지를 한데 묶는다. 첫째, 레이크하우스는 재료 창고다.[page:3] 둘째, 프롬프트 정제 및 AI 런타임은 조리 엔진이다.[page:2] 셋째, 카탈로그와 대시보드는 이미 조리된 레시피를 진열하고 바로 꺼내 쓰게 하는 사내 마켓플레이스다.[web:23][web:113]

## 제품 원칙

- **Search before build**: 새 프롬프트나 새 에이전트를 만들기 전에 유사 예제를 먼저 찾는다.[page:2][web:23]
- **Prompt as asset**: 프롬프트는 일회성 입력이 아니라 버전 관리되는 자산으로 다룬다.[page:2]
- **Multimodal by default**: 응답은 텍스트만이 아니라 표, CSV, 이미지, SVG, 2D/3D 차트까지 포함할 수 있어야 한다.[web:119]
- **Dashboard first UX**: 모든 주요 작업은 대시보드 내 패널, 모달, 사이드 롤업/드로어에서 빠르게 수행 가능해야 한다.[web:113][web:125]
- **Governed autonomy**: 에이전트는 자율적으로 보이더라도 권한, 승인, 감사 경계를 벗어나면 안 된다.[web:35]

## 배경과 문제 정의

기업은 생성형 AI를 도입할 때 보안, 데이터 단절, 결과 재현성 부족, 중복 개발, 그리고 운영 가시성 부족 문제를 동시에 겪는다.[page:1][page:2][web:35] 내부 데이터 기반으로 안전하게 AI를 활용하고 비전문가도 손쉽게 에이전트를 만들게 하는 방향은 이미 AIU 사례에서 확인되지만, 실제 제품화 단계에서는 “어떤 프롬프트와 에이전트가 이미 있는가”, “무엇을 재사용할 수 있는가”, “출력 결과를 어떤 형태로 업무에 바로 쓸 수 있는가”가 중요해진다.[page:1][web:23]

또한 현대 엔터프라이즈 AI는 텍스트 답변만으로는 부족하다. 사용자는 표 형태 결과, 다운로드 가능한 CSV, 이미지 결과, SVG 다이어그램, 2D/3D 차트 같은 구조화된 산출물을 기대하며, 이는 대시보드형 UI와 잘 결합될 때 업무 도구로서 가치가 커진다.[web:112][web:119] 따라서 Forgenta는 챗 중심 인터페이스를 유지하되, 멀티모달 출력과 앱형 실행 환경을 기본 기능으로 포함해야 한다.[web:112][web:123]

## 제품 목표

### 비즈니스 목표

- 내부 데이터 기반 생성형 AI 활용 표준 환경 구축.[page:1]
- 제조, HR, finance 도메인에서 재사용 가능한 앱/에이전트 자산 축적.[page:3][web:23]
- 카탈로그 기반 탐색과 복제로 중복 개발 감소.[page:2][web:23]
- 거버넌스와 감사 가능한 엔터프라이즈 AI 운영 체계 확보.[web:35]
- 멀티모달 출력과 대시보드 UX를 통해 실제 업무 사용성을 확보.[web:112][web:119]

### 사용자 목표

- 사용자는 내부 문서와 데이터에 근거한 답변을 받을 수 있어야 한다.[page:1][page:3]
- 사용자는 프롬프트를 자동 정제받고 유사 예제를 추천받아야 한다.[page:2]
- 사용자는 에이전트 또는 앱을 카탈로그에서 선택해 사용, 편집, 삭제, 이동할 수 있어야 한다.[web:113][web:125]
- 사용자는 채팅 결과를 표, CSV, SVG, 이미지, 2D/3D 차트 형태로 받을 수 있어야 한다.[web:119]
- 사용자는 깔끔한 대시보드에서 챗, 비교, 결과, 프로필, 시스템 메뉴를 일관되게 사용할 수 있어야 한다.[web:112][web:123]

- 모바일 앱 네이티브 버전.

## 대상 사용자

### 1. 일반 사용자

챗과 카탈로그를 통해 필요한 앱/에이전트를 찾아 실행하고 결과를 다운로드하는 사용자다. 이 사용자는 “직접 설계”보다 “찾아서 사용” 경험을 더 자주 쓴다.[web:23]

### 2. 시민개발자 / 파워 유저

기존 에이전트나 앱을 복제한 뒤 프롬프트, 데이터 소스, 출력 형식을 수정해 자신의 팀용으로 재구성하는 사용자다.[web:23][web:25] 이 사용자는 프롬프트 비교, 설정 편집, 결과 검증 기능의 혜택을 크게 받는다.[page:2]

### 3. 관리자 / 운영자

카탈로그 관리, 권한 정책, 데이터 소스 연결, 배포 승인, 감사 로그 검토를 담당하는 사용자다.[web:35] 이 사용자는 삭제, 이동, 공개 범위 변경 같은 자산 수명주기 작업을 수행해야 한다.[web:113]

## 핵심 사용자 시나리오

### 시나리오 A: 챗으로 질문하고 멀티모달 결과 받기

사용자는 제조 품질 이슈나 HR/finance 데이터를 질문한다. 시스템은 자연어 답변뿐 아니라 필요한 경우 표, CSV 다운로드 링크, 이미지 미리보기, SVG 다이어그램, 2D/3D 차트를 함께 반환해야 한다.[web:119]

### 시나리오 B: 기존 앱/에이전트를 카탈로그에서 찾아 실행하기

사용자는 대시보드 카탈로그에서 “비용 이상치 분석 앱” 또는 “정비 보고서 생성 에이전트”를 찾는다. 상세 보기를 열고 즉시 실행하거나, 자신이 가진 권한 범위 안에서 편집 또는 복제를 시작할 수 있어야 한다.[web:23][web:25]

### 시나리오 C: 프롬프트를 정제하고 유사 예제 비교하기

사용자가 자유 입력으로 질문하면, 시스템은 구조화된 정제 프롬프트를 생성하고 유사 프롬프트와 예제를 추천한다.[page:2] 사용자는 원문과 정제본을 비교하고, 기존 예제와 어떤 차이가 있는지 본 뒤 실행을 결정할 수 있어야 한다.

### 시나리오 D: 카탈로그에서 삭제·이동 작업 수행하기

관리자 또는 권한 있는 사용자는 카탈로그 항목을 삭제하거나, 특정 보드/폴더/카테고리로 이동해야 한다. 이 작업은 모달 또는 사이드 패널에서 수행 가능해야 하며, 대시보드 편집 UX의 일반 패턴처럼 명확한 확인 절차를 제공해야 한다.[web:113][web:114][web:125]

## 기능 요구사항

### 1. 브랜드 및 시작 화면

- 제품명은 모든 UI와 문서에서 **Forgenta: Hybrid Agentic AI - App Platform**으로 표기한다.
- Splash 화면에는 로고, 짧은 제품 설명, 로그인 옵션을 제공한다.
- 로그인 옵션은 Google, Microsoft, 기업 SSO, 이메일 로그인 4가지를 기본 제공한다.[web:120][web:117]
- 기업 환경을 고려해 SSO는 SAML/OIDC 확장 가능 구조를 전제로 한다.[web:117]

### 2. 사내 대시보드 UI

- 챗봇 UI.
- 프롬프트 원문 vs 정제본 비교 패널.
- 멀티모달 결과 뷰어.
- 카탈로그 진입 및 최근 사용 항목 표시.
- 로그인 사용자 프로필 메뉴, 워크스페이스 선택, 시스템 메뉴.
- 깔끔한 정보 밀도의 대시보드 스타일 적용.[web:112][web:123]

### 3. 카탈로그 관리 UI

개발된 앱 혹은 에이전트를 카탈로그 형태로 보여주고, 선택 시 사용, 편집, 삭제, 이동이 가능한 UI를 대시보드에 포함해야 한다. 대시보드 편집 및 사이드 패널 액션 패턴은 Looker와 Liferay 문서에서도 일반적인 관리 UX로 확인된다.[web:113][web:125]

필수 기능은 다음과 같다.

- Agent Catalog와 App Catalog 분리 제공.[web:23]
- 카드 뷰와 리스트 뷰 지원.
- 항목 클릭 시 상세 보기 패널 표시.
- 상세 보기에서 Use, Edit, Clone, Delete, Move 액션 제공.[web:25][web:113]
- Delete는 별도 확인 모달 필수.[web:114]
- Move는 드로어/사이드 패널에서 목적지 선택 방식 지원.[web:113][web:125]
- Edit는 롤업 패널 또는 전체 편집 화면으로 진입.
- 권한 없는 사용자는 Delete/Move/Edit 비활성화.

### 4. 프롬프트 정제 및 비교

- 자유 입력 프롬프트 정제.
- 의도 분류, 구조화 정규화, 보안 필터, 누락정보 탐지.[page:2]
- 원문과 정제본 diff 비교 뷰.
- 유사 프롬프트 및 유사 예제 추천.
- 프롬프트 저장, 버전 관리, 평가 기록.[page:2]

### 5. 멀티모달 응답 엔진

Forgenta는 챗/프롬프팅 결과를 텍스트 외의 다양한 형태로 반환해야 한다. 데이터 분석 및 대시보드 문맥에서 2D/3D 시각화와 구조화 결과 지원은 웹 기반 데이터 도구의 중요한 기능으로 널리 사용된다.[web:119]

필수 출력 타입은 다음과 같다.

| 출력 타입 | MVP 요구사항 |
|---|---|
| Text/Markdown | 기본 답변, 설명, 근거 요약 |
| Table/Grid | 정형 결과 테이블, 정렬/스크롤 가능 |
| CSV | 생성 후 다운로드 가능 |
| SVG | 생성 및 뷰잉 지원, 외부 서비스/라이브러리 연동 여부 기술 검토 필요 |
| Image | 이미지 미리보기, 확대, 다운로드 |
| 2D Chart | bar, line, pie, scatter, area |
| 3D Chart | 3D scatter, 3D surface, 3D bar 유사 시각화 |

추가 요구사항은 다음과 같다.

- 이미지 결과는 뷰어와 다운로드 버튼 제공.
- CSV 결과는 다운로드 가능해야 한다.
- SVG는 렌더링 실패 시 raw markup 또는 fallback preview를 제공해야 한다.
- 차트는 대시보드 패널 안에서 확대/축소 또는 새 창 보기 지원.
- 2D/3D 차트는 라이브러리 기반 렌더링을 전제로 한다.[web:119]

### 6. 데이터 레이어

버전 1.2의 데이터 레이어는 오픈·공유 제조업 데이터와 synthetic HR, synthetic finance 데이터를 저장하는 데이터 레이크하우스를 포함해야 한다.[page:3] 레이크하우스는 구조화, 반정형, 비정형 데이터를 한 환경에서 저장·관리하고 거버넌스를 제공하는 기반으로 사용한다.[page:3]

저장 구조는 Bronze, Silver, Gold 3계층을 권장한다.[page:3]

| 계층 | 목적 |
|---|---|
| Bronze | 원천 적재, 원본 보존 |
| Silver | 정제, 표준화, 품질 검증 |
| Gold | 분석/앱 서비스 최적화, RAG용 curated 데이터 |

## 기술 검토 포인트

### SVG 생성

SVG 생성은 두 가지 방식이 가능하다. 첫째, 모델이 직접 SVG 마크업을 생성하도록 하는 방식이다. 둘째, 별도 렌더링 또는 다이어그램 라이브러리와 연동해 구조화된 입력에서 SVG를 생성하는 방식이다. MVP에서는 모델 직생성 + 렌더링 검증 방식으로 시작할 수 있으나, 복잡한 도식이나 안정적 렌더링이 필요하면 외부 서비스 또는 전용 다이어그램 엔진 연동 검토가 필요하다.

### 2D/3D 차트

브라우저 기반 데이터 시각화 생태계에는 Apache ECharts, Plotly 등 2D/3D 렌더링이 가능한 도구가 존재하므로, MVP에서도 라이브러리 기반 구현이 현실적이다.[web:119] 3D 차트는 성능 부담이 있으므로 우선 scatter/surface 위주로 제한하는 것이 적절하다.[web:119]

### 인증

Google, Microsoft, 이메일 로그인과 기업 SSO를 함께 지원하는 구조는 엔터프라이즈 제품에서 일반적인 요구이며, SAML/OIDC 확장 구조를 사전에 잡아두는 것이 바람직하다.[web:117][web:120]

## 최종 정리

Forgenta v1.2는 단순한 AI 챗 도구가 아니라, **카탈로그 기반 재사용**, **멀티모달 출력**, **대시보드형 운영 UX**, **엔터프라이즈 인증 및 거버넌스**를 갖춘 MVP 수준의 하이브리드 에이전틱 AI 앱 플랫폼으로 정의된다.[web:23][web:35][web:120] 이 정의를 기준으로 다음 단계에서는 화면 IA, 시스템 아키텍처, API 명세, 그리고 MVP 개발 백로그를 바로 도출할 수 있다.[web:112][web:113]
